# 所有 vs アクセス軸

道具やサービスを評価するとき、「自分が所有するか／貸主のサービスにアクセスするか」という軸。コンプライアンス要件・データ管理責任・事業継続性を持つ事業者には決定的に効くが、それらを持たない事業者には可視化されない、文脈依存的な評価軸。

## 起源（Why this exists）

道具の所有・占有は、近代以前は物理的事実だった。土地、家畜、工具、書物。道具は手元にあり、貸主は領主か質屋程度。

産業化以降、道具のサービス化が進む。レンタル、サブスクリプション、SaaS、クラウド、API。「道具を借りる」が経済の中心になる。

借りている道具は、貸主の都合で消える。これは契約構造の必然であり、技術の性質ではない。Apache 2.0 のように撤回不能な許諾もあれば、利用規約一行で全機能停止できるサービスもある。

「所有 vs アクセス」が評価軸として機能するかどうかは、業務側の文脈に依存する：

- 個人情報・契約情報を扱う業務 → 軸が立つ
- 監査要件のある業務 → 軸が立つ
- 単発・低リスクの趣味業務 → 軸は無関係

## 観察可能なproxy

- ライセンスの永続性（OSS、特に Apache 2.0／MIT／BSD vs プロプライエタリ）
- 契約上の規約変更余地（一方的変更条項の有無）
- データ越境の有無（サーバ所在地、サブプロセッサの開示）
- 監査ログの可視性（誰が何を処理したか追跡可能か）
- 運用継続性（提供者が消えても道具が残るか）

## 介入点（Disable conditions）

このatomは「軸」なので「打ち消す」対象ではない。代わりに：

- **コンプライアンス要件の特定**: Pマーク、宅建業、医療系等の規制要件を業務に紐づける
- **データフロー可視化**: 「どのデータがどこを通るか」を意思決定前に図示する
- **撤回不能性の要求**: ライセンス・契約条件に「規約変更不可」「ライセンス撤回不能」を要求する／なければ候補外にする

## 補足

programmability-as-axis と並ぶ「業務文脈で初めて意味を持つ軸」atom。両者は道具選定における二大「隠れ軸」として機能する。

ownership-vs-access は **独立性** を、programmability-as-axis は **量産性** を、それぞれ可視化する。

## 哲学的親族

- **マルティン・ハイデガー『存在と時間』(1927) §15-16**: Zuhandenheit（手許性）と Vorhandenheit（眼前性）。所有 ≒ Zuhanden（手許に溶け込む道具）、アクセス ≒ Vorhanden（対象として向き合うサービス）の対応がある
- **マックス・ヴェーバー『支配の社会学』(1922)**: 鉄の檻。アクセス側に立つことは、貸主の合理化機構に組み込まれることである
