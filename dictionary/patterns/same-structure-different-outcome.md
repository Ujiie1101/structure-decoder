# Same-Structure Different-Outcome Pattern（同構造・異結果比喩パターン）

## 定義
論点と**同じ構造を持つ日常例**を持ち出し、両者の結果が異なることを示したうえで、「違いは何か」を1変数で断定するリフレーミング手法。

## 公式
```
論点A: 構造X → 結果Y（否定的/炎上）
日常例B: 構造X → 結果Z（許容/当然）
↓
「同じ構造なのに、なぜ結果が違うのか？」
↓
「違いは[1つの変数]だけ」
↓
問題の真の原因が再定義される
```

## メカニズム

### 1. 抽象→日常のドロップ
抽象的な構造論は理解されにくい。日常例に落とすと一瞬で伝わる。

### 2. 対比による論点の純化
同じ構造なのに結果が違う→ **違いはどこにあるか**に読者の注意が集中する。

### 3. 1変数への絞り込み
「違いは複数ある」ではなく「違いは○○だけ」と断定することで、問題の本質が見える。

## 成功条件
- 日常例が**読者全員が知っている**ものである
- 構造が本当に同一である（こじつけは即バレる）
- 違いを**1つの変数**に絞れる

## 失敗パターン
- 日常例がマニアック → 伝わらない
- 構造が実は違う → 論理破綻
- 違いが複数 → 問題が曖昧化

## 事例

### Day 25「99万人の沈黙」
```
論点: SNSで「世界平和」と書く → 炎上
日常例: コンビニ募金箱に10円入れる → 許容（年間20億円）
↓
同じ「平和を願う小さな善意」
↓
違いは「可視性」だけ
↓
問題は「願い」ではなく「見えること」にあると再定義
```

### 他の応用候補
- 「飲食店での会話」vs「SNSでの会話」→ 違いは記録性
- 「手紙の苦情」vs「SNSの苦情」→ 違いは拡散性
- 「日記の怒り」vs「投稿の怒り」→ 違いは到達範囲

## 適用条件
- 論点が「なぜこれだけ炎上/批判されるのか」という問いを含む
- 類似構造の日常例が見つかる
- 違いを言語化できる

## 配置場所
- 記事の**後半（70〜80%地点）**が最適
- 既に事例・構造が提示された後
- 読者が「構造はわかったが納得しきれない」状態での最後の一押し

## 注意点
- 日常例を**1つに絞る**（複数出すと論点分散）
- 「違いは○○だけ」は**太字**で視認性確保
- 皮肉・誇張は避ける（真摯な問いとして提示）
