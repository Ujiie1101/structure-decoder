# Methodology — structure-decoder の方法論

structure-decoder は「テキスト・状況を構造で読む」装置です。本ドキュメントは、その背後にある原理・辞書設計・エンジン挙動・拡張方法を説明します。

---

## 1. 何を解決しているか

generic な大規模言語モデル（LLM）は「この文章を要約して」には答えますが、「この文章にどの構造パターンが起動しているか」には答えられません。

理由は単純です：**事前辞書がないから**。

LLM は無限の事例を持っていますが、「これは A パターン、これは B パターン」と分類するための**安定した語彙**を持ちません。質問のたびに、毎回ゼロから「構造的な特徴」を生成します。それは「印象批評」と区別がつきません。

structure-decoder は、数百冊の読書と数百日の観察から抽出された**事前辞書**を装置に組み込むことで、この問題を解決します。同じ入力に対して、同じ辞書を引けば、同じ構造判定が出る——という再現性を担保します。

> **構造分析の核心は「速さ」ではなく「再現可能な語彙」である**。

---

## 2. 核心原理：4層分解

structure-decoder の中心にあるのは、**1つの load-bearing 文（または段落・状況）を 4 つの層に分解する**という操作です。

```
入力（一文 / 段落 / 状況）
  ↓ 何を事実として述べているか
§1 事実層（denotative）
  ↓ どんな修辞的選択がなされているか
§2 修辞層（rhetorical）
  ↓ 著者が暗黙に主張している命題は何か
§3 構造命題層（structural proposition）
  ↓ 書物全体／状況全体にどう接続するか
§4 接続層（book-wide / context-wide connection）
```

### なぜ 4 層か

3層では「事実 → 修辞 → 命題」だけで止まり、**その命題が個別の作品／状況にどう接続するか**が見えません。一方、5層以上に細分化すると焦点が消えます。

「**4層**」は経験的に最も再現性が高い分割でした。試行錯誤の結果としての慣例値であり、将来的に分割が変わる可能性は残します（実際 v0.5 で 5 番目に近い「執筆背景レイヤー」を追加しています）。

### 各層の役割

| 層 | 問い | 出力の質 |
|---|---|---|
| **§1 事実層** | この文／状況は文字通り何を述べているか？ | 反論不可能な中立記述まで還元 |
| **§2 修辞層** | なぜこの語・この構文・この順序か？ | 語ごとの差分分析（他の語ならどう変わるか） |
| **§3 構造命題層** | 著者／当事者が暗黙に主張しているのは何か？ | 反証条件付きの命題、最大3個 |
| **§4 接続層** | 全体（書物 / プロジェクト / 関係）にどう繋がるか？ | 章・人物・後続フェーズへの伏線関係 |

§2 と §3 が肝です。§2 で語のミクロな選択を見ず、§3 でいきなり「テーマ」を抽出する読みは、印象批評に堕ちます。

---

## 3. 二つの読みのレンズ

4 層分解、特に §3（構造命題層）の抽出は、**二つのレンズを通して**初めて再現性を持ちます。

### レンズ 1：人文知の背景

§3 を取るとき、読み手は**どの人文知の伝統で読んでいるか**を宣言します。

```
主レンズ: ブルデュー資本論（ハビトゥス継承）
副レンズ: ゴフマン相互行為論（自己呈示）
採用理由: 本文が「親譲り」と血統的継承を扱い、かつ自嘲という呈示形式を取るため
```

これにより：
- 読みが**反証可能**になる（別レンズなら別の命題が出るはず）
- 命題が**既存の人文知に位置づけ**られる（孤立した発見ではなくなる）
- 「採用理由を書けないレンズ」は外せる（思いつき重ねの防止）

### レンズ 2：構造フィルター

`dictionary/atoms/` と `dictionary/patterns/` の辞書を、**能動的フィルター**として本文に当てます。

```
| pattern/atom              | 適用結果 | コメント |
|---------------------------|---------|---------|
| mission-drift             | △ 部分  | 「親譲り」が原点として機能 |
| founders-curse            | ❌ 通らない | 個人主体ではなく血統主体 |
| self-deprecation-as-...   | ✅ 通る | 新atom候補 |
```

**通らなかったフィルターを必ず記録します**。これが「この本文が何ではないか」の輪郭を作り、固有性を見えるようにします。

### なぜ両輪なのか

- 構造フィルター単独 = 機械的なマッチング。意味付けが薄い
- 人文知レンズ単独 = 解釈学的だが、再現性がない
- **両輪が回って初めて「再現可能な構造分析」になる**

人文知が**意味付け**を、構造フィルターが**形式判定**を担います。

## 4. 辞書の三層構造

`dictionary/` 以下は、粒度の異なる 3 つの層に分かれています。

```
dictionary/
├── atoms/         # 構造原子（79件）   = 分解不可能な最小単位
├── patterns/      # 構造パターン（232件）= atom の組み合わせで生まれる再現可能な構造
├── mechanisms/    # OS層メカニズム（34件）= パターンを跨いで作用する civilization-level の動因
└── meta/          # メタ層 = 書物全体や複数 entity を跨ぐ技法
```

### Atom（構造原子）

「これ以上分解すると構造が消える」最小単位の動作・関係です。

例:
- `context-stripping`：制度を通過する過程で情報のコンテキストが構造的に削ぎ落とされる動作
- `addressee-postvariation`：送り手が決まった後で受け手側が変動することによる意味変容
- `calibrated-audibility`：聞こえる音量を自分の権力位置に応じて調整する習慣

Atom は **「動作」または「関係」の名前**として記述されます。「現象」や「結果」ではなく、**動詞的なもの**。

### Pattern（構造パターン）

複数の atom が組み合わさって生まれる、**再現可能な複合構造**です。

例:
- `inquiry-foreclosure`（問いの先回り閉鎖）= 知らないものに出会ったとき、問いが立つ前に既知の箱に分類してしまう構造
- `prevention-self-erasure`（予防の自己消去）= 防いだ被害が反実仮想でしか語れず、便益が不可視化される構造

各 pattern は frontmatter に `composition:` を持ち、構成 atom を明示します。

### Mechanism（OS層メカニズム）

複数のパターンを跨いで作用する、**civilization-level の動因**です。

例: `cluster-empire`, `cluster-collapse`, `archetype-registry`

mechanism は個別の事象を超えて、文明・組織・歴史の長周期で観察される動作を捉えます。

### Meta（メタ層）

書物全体や複数 entity を跨ぐ技法。例: `chapter-pivot-line`, `title-self-inversion`

## 5. エンジン群

`dictionary/` を入力として動かすのが、`.claude/skills/` に置かれた engine 群です。Claude Code がこのリポジトリで起動されると、自動的に読み込まれます。

### `/book-decompose`（テキスト入力）

書物の一文・段落・章を入力すると、4層分解 + KOS 候補（atom/pattern）+ 新造語候補を出力します。

6 つのモード：

| Mode | 用途 |
|---|---|
| 1. Decompose | 単文を 4 層分解 |
| 2. Extract | 段落から load-bearing 文を抽出 |
| 3. Article | 古典×現代の記事執筆 |
| 4. International | 海外ブログ向け英訳版 |
| 5. Reading Note | 内部用 reading-note（観察者の独白） |
| 6. Published | note 公開用 ※注釈形式 |

実例: `examples/henshin/chapter-1-reading-note.md`, `examples/botchan/published/chapter-1.md`

### `/atom-decompose`（パターン入力）

既存パターンを atom に分解し、**未試行の atom 組み合わせから新パターンを演繹**します。

```
/atom-decompose <pattern-id>     # 既存patternを分解
/atom-decompose --derive          # 新pattern演繹（未試行組合せ）
/atom-decompose --article <pat>   # atom分解型記事を執筆
```

これは Knowledge OS の自己拡張装置です。辞書を引くのではなく、辞書を増やします。

### `/diagnose`（状況入力）

自分の判断・組織・関係性・進行中プロジェクトを「状況」として入力すると、`dictionary/patterns/` 全件に照合して構造判定 + 打てる手を出力します。

出力 4 種：

| 判定 | 内容 |
|---|---|
| GO | 続行。次の閾値まで様子見 |
| HOLD | 一時停止。観測指標を増やしてから再判定 |
| PIVOT | 方向転換。同じ目的を別の構造で達成 |
| EXIT | 撤退。これ以上のコストは沈む |

**撤退ラインを書かない診断は無効**——同情モード禁止、構造として何が起きているかだけを書きます。

### 3つの入力チャネルの構造

```
[書物・記事]     ──→ /book-decompose ──→ 4層 + atoms/patterns 候補
[既存パターン]   ──→ /atom-decompose ──→ atoms 分解 + 新pattern演繹
[個人の状況]     ──→ /diagnose      ──→ GO/HOLD/PIVOT/EXIT + 打てる手
                          ↓
              共通: dictionary/ を辞書として参照
```

3 つは別チャネルですが、**同じ辞書を引きます**。これが再現性の源泉です。

## 6. 拡張方法

### 新しい atom を追加するとき

以下の条件を満たすときに atom 化します：

1. **これ以上分解すると構造が消える**ことが確認できる
2. **複数の文脈で再現観察**される（少なくとも 3 つ以上の異なる場面）
3. 既存 atom に**包含されない**（独立性チェック済み）

新 atom ファイルのテンプレート：

```yaml
---
id: "your-atom-id"
name: "日本語名"
type: "atom"
domain: "institutional | cognitive | relational | etc"
created: YYYY-MM-DD
derived_from:
  - text: 出典書名
    author: 著者
    year: 年
    chapter: 場所
---

# {atom-id}（日本語名）

{1-2段落で動作・関係を記述}

## 起源（Why this exists）
{なぜこの動作が成立するか、3つの構造的要因}

## 構成
- {atom が組み合わさる場合は明示}

## 関連
- [[other-atom-id]]
- [[some-pattern-id]]
```

### 新しい pattern を追加するとき

atom が複数組み合わさって**再現可能な複合構造**が見えたときに pattern 化します。

```yaml
---
id: "your-pattern-id"
name: "日本語名"
grammar_type: "process | erosion | inversion | etc"
mechanism_id: "E-NNN"
created: YYYY-MM-DD
source: "観察元の出典"
composition:
  - atom: existing-atom-1
    role: "基本層：〜"
  - atom: existing-atom-2
    role: "時間層：〜"
related:
  - related-pattern-1
---

# {pattern-id}（日本語名）

{パターンの定義、起動条件、典型事例}
```

### Pull request

新規追加・修正は GitHub の Pull Request で歓迎します。基本方針：
- 出典（`derived_from`）必須
- 既存 atom / pattern と**独立**であることの確認
- 最低 3 場面での観察例（コメントに記載）

## 7. このプロダクトの限界

### 何**ではない**か

- ❌ 要約装置（要約は LLM 単独でもできる）
- ❌ データベース検索（パターン名で grep するだけなら git でできる）
- ❌ 厳密に決定論的（同じ入力で完全に同じ出力が出るとは限らない）

### 何**である**か

- ✅ 構造的読みのための事前辞書
- ✅ 印象批評を抑止するフィルター
- ✅ 「通らないフィルター」を可視化することで固有性を見せる装置
- ✅ 再現性のある分析語彙の供給源

### 限界

1. **辞書は日本語ベース**：海外展開には翻訳が必要
2. **辞書は作者の読書履歴に基づく**：偏りがある（西洋哲学・社会学・日本近代文学が厚い）
3. **新分野の atom はまだ薄い**：自然科学・数学・建築などはほぼ未着手
4. **判定は確率的**：同じ入力でも、レンズ宣言が変われば命題は変わる

これらは「不完全さ」ではなく「**設計上の選択**」です。完全網羅を目指さず、**通らないフィルターを記録する**ことで、辞書の輪郭そのものを毎回テストします。

## 8. 再現性について

structure-decoder の主張は「**再現可能な構造分析**」です。ここで言う再現性は、自然科学的な決定論ではなく、以下のことを指します：

1. **同じ辞書**を引けば、同じ atom/pattern の候補が出る
2. **同じレンズ**を宣言すれば、同じ命題が出やすい
3. **通らないフィルター**は、読み手によらず同じものが通らない

つまり「読み手によって命題が変わる」という解釈学的余地は残しますが、「**何がフィルターを通り、何が通らないか**」は読み手によらず一定です。

この**判定の中間状態**を可視化することが、本プロダクトの再現性の正体です。

## 10. さらに読むには

- 各 atom / pattern ファイル：`dictionary/` 配下を直接読むのが最短
- 動作実例：`examples/henshin/`, `examples/botchan/`, `examples/shayo/`
- スキル仕様：`.claude/skills/{book-decompose,atom-decompose,diagnose}/skill.md`

Christopher Alexander の『パタン・ランゲージ』（1977）が建築でやったことを、テキストと状況の領域で——というのが本プロダクトの位置づけです。
